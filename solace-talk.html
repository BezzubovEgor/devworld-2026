Слайды 6–8

Решение: Event-Driven Architecture

⚡

Async Communication

Оркестратор отправляет задачу → продолжает работу → получает результат когда подагент закончил. Long-running workflows на часы — тривиальны.

📡

Pub/Sub Messaging

Агенты публикуют в топики и подписываются. Не знают друг о друге. Новые агенты добавляются без перенастройки.

🔴

Real-Time Streams

Агенты получают операционные данные в момент события — inventory changes, транзакции, alerts. Не polling, а push.

🛡️

Fault Tolerance

Буферизация событий, dead letter queues, изоляция плохих агентов, rate limiting зацикленных.

Deep Dive

Solace Agent Mesh — как это работает

Open-source фреймворк, где агенты общаются асинхронно через Solace Event Broker, обмениваются артефактами и делегируют задачи через стандартизированный протокол discovery.

🎭 Orchestrator Mode

LLM-powered координатор. Получает задачу → reasoning над доступными агентами → динамически решает кого вызвать и в каком порядке. Для open-ended задач: «проанализируй датасет», «напиши ответ клиенту».

⚠️ Проблема: LLM — это probability distribution. Один и тот же промпт → разные результаты. Для compliance и аудита это неприемлемо.

⚙️ Workflows Mode

Explicit control flow — фиксированные шаги в фиксированном порядке. Для детерминистичных процессов: RFQ, compliance flows, аудитируемые пайплайны.

✅ Гарантия: Каждый шаг задокументирован, порядок фиксирован, есть audit trail. Идеально для regulated industries.

🧪 Built-in Eval Framework

«Seeming to work и actually working reliably — это очень разные вещи». Solace Agent Mesh включает framework для тестирования агентов:

Test Cases — JSON-сценарии с ожидаемым поведением Test Suites — группировка сценариев Real Broker — тесты на реальном брокере, не моки OpenTelemetry Metrics — мониторинг latency/throughput

Проблемы Production

Почему чистый LLM-orchestrator ломается в проде

Solace явно выделяет failure modes, которые мешают enterprise adoption.

🎲

Non-determinism

LLM сэмплирует из probability distribution. Temperature и prompt engineering сужают, но не устраняют variance.

🔍

Observability Gaps

Когда оркестратор принимает неверное решение — нет stack trace, нет line number, нет детерминистичного условия для дебага.

📐

Schema Drift Brittleness

Агент меняет формат выхода → LLM «адаптируется» молча, маршрутизируя данные не тому агенту без ошибки.

📋

Compliance & Audit

«LLM решил шаги в runtime» — это не audit trail. Regulated industries требуют доказуемый порядок выполнения.

💸

Runaway Costs

Плохо ограниченный loop — LLM повторно делегирует задачу, которую не может решить. Сжигает токены без circuit breaker.

🔄

Silent Degradation

Model updates, config drift, tool changes — меняют поведение агента без видимых breakages. Без eval-фреймворка не поймать.

⚡ Релевантность для Kineto

1

Kineto уже решает часть этого — оркестрация агентов, queue modes (collect/steer/followup), concurrency control

2

Pub/sub паттерн — стоит подумать для multi-agent: event bus вместо прямых вызовов

3

80/20 айсберг — отличная метафора для маркетинга: «мы решаем 80% подводной части»

4

DLQ для агентов — dead letter queue для failed agent tasks стоит добавить

5

Google ADK — Solace строит на ADK, Kineto в экосистеме Google → потенциал совместимости

6

Orchestrator vs Workflows — двойной режим (LLM-driven + deterministic) — полезный паттерн. Kineto queue modes уже частично это делают

7

Eval Framework — встроенное тестирование агентов на реальном брокере. «Vibes → Evidence» — применимо к любой агентной платформе
